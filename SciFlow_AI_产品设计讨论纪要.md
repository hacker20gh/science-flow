# SciFlow AI — 产品设计讨论纪要

> **日期**：2026-07-10
> **阶段**：概念设计 v1
> **状态**：方向确认，待深入细化

---

## 一、产品定位

**SciFlow AI**：AI 驱动的科研全流程工作流网站，覆盖从文献调研到实验设计到论文发表的完整链路。

### 核心理念

不是三个独立工具的拼盘，而是一条**有记忆的循环式工作流**——每个阶段的输出自动成为下一个阶段的输入，失败和回溯都是流程的正常组成部分。

**一句话定位**：

> SciFlow 是一个陪你螺旋上升的科研伙伴。每一圈回来，它都比上一圈更懂你的课题。

---

## 二、初始构想（原始方案）

用户提出三个功能板块：

| 板块 | 核心功能 |
|------|---------|
| **实验设计** | 步骤规划、浓度稀释计算、对照组检查、靶点推荐 |
| **课题调研** | 智能文献检索、提取药物/通路/表型信息、生成机制矩阵 |
| **论文写作** | 学术英语润色、数据转 LaTeX 格式 |

---

## 三、评估与迭代

### 3.1 对三个板块的评价

| 板块 | 评价 | 建议 |
|------|------|------|
| **课题调研（机制矩阵）** | ⭐ 最有价值、最具创新性 | 核心卖点，优先实现。"比较十篇"比"读懂一篇"更接近真实需求 |
| **实验设计** | 方向正确，但不应是独立工具 | 应该是文献调研的自然延续——机制矩阵 → 自动推荐实验方案 |
| **论文写作** | 红海领域（Paperpal、Writefull 等已有产品） | 差异化定位：不是润色，是"从已有积累一键组装草稿" |

### 3.2 补充的功能想法

| 功能 | 价值 | 优先级 |
|------|------|--------|
| **Research Memory（课题记忆）** | 用得越久越精准，是长期护城河 | 高（架构层面预留） |
| **数据到图表的智能转换** | 原始数据 → 统计分析 → publication-ready 图表 | 高 |
| **Reviewer Simulator（审稿人模拟）** | 投稿前模拟同行评审，预判修改意见 | 中 |
| **实验排障诊断（Troubleshooting）** | 实验失败时基于已知模式诊断原因 | 高（核心体验） |
| **Gap Finder（空白发现）** | 机制矩阵中的空白处 = 潜在创新点 | 高（机制矩阵子功能） |

### 3.3 机制矩阵的关键设计追问

| 问题 | 建议方案 |
|------|---------|
| 文献结论互相矛盾怎么办？ | 加"一致性评分"或"争议标记" |
| 不同期刊的结论权重相同吗？ | 按 IF、样本量、实验方法分层（证据强度） |
| 矩阵留白处有没有价值？ | Gap Finder 高亮——直接指向创新点和选题方向 |

## 四、工作流设计演进

### 4.1 第一版：线性管线（已否决）

```
探索 → 验证 → 输出
```

**问题**：现实中实验不会那么顺畅，用户必然会在某个阶段回溯。线性模型不反映真实科研过程。

### 4.2 第二版：循环式工作流（确认方向）

```
                    ┌──────────────────────────────┐
                    │                              │
                    ▼                              │
               ┌─────────┐                         │
               │  探索     │←── 结果矛盾 ───┐       │
               │  Explore │                 │       │
               └────┬────┘                 │       │
                    │                      │       │
                    ▼                      │       │
               ┌─────────┐                 │       │
               │  验证     │──→ 意外结果 ───┘       │
               │  Verify  │                        │
               └────┬────┘                        │
                    │                              │
                    ▼                              │
               ┌─────────┐                        │
               │  输出     │──→ 发现缺实验 ←────────┘
               │  Output  │
               └─────────┘
```

**核心设计原则**：

| 原则 | 说明 |
|------|------|
| **循环而非线性** | 探索→实验→结果→可能回到探索，每圈都积累 |
| **失败是一等公民** | 记录、分析、引用，不隐藏 |
| **回溯带上下文** | 不是回到空白页，是带着"为什么回来"的诊断 |
| **知识面板实时更新** | 机制矩阵和假设状态始终反映最新认知 |
| **时间线记录全部** | 包括失败、转向、反复——写论文时都有用 |
| **阶段可任意跳转** | 底部快捷操作，不需要按顺序来 |

### 4.3 回溯场景覆盖

| 现实场景 | 用户需要做什么 | SciFlow 怎么支持 |
|---------|-------------|-----------------|
| 实验彻底失败（污染、细胞死） | 排障、换方案 | 记录失败原因，推荐排查方案，带着失败信息重新设计 |
| 结果跟预期矛盾 | 回去查文献、换假设 | 定向搜索"解释这个矛盾"，而非泛搜 |
| 导师说方向不对 | 保留已有积累，转新方向 | 分支（branch）：旧方向保留在时间线，新方向从这里分叉 |
| 写到一半发现缺对照实验 | 停笔，回去补实验 | 从输出直接跳到验证，带着"缺什么"的上下文 |
| 新文献出来了 | 更新认知，调整方案 | 补充进机制矩阵，系统标注对当前假设的影响 |

## 五、界面架构

### 5.1 整体布局：时间线 + 知识面板 + AI 助手

```
┌──────────────────────────────────────────────────────────────┐
│  课题：PD-1 耐药机制                              [AI助手 🤖] │
├────────────────────────────────┬─────────────────────────────┤
│                                │                             │
│   📅 项目时间线                 │   🧠 课题知识面板             │
│                                │                             │
│   每个事件是一个节点：           │   ┌─ 机制矩阵（自动更新）──┐  │
│   📖 文献添加                   │   │ ...                    │  │
│   💡 假设提出                   │   └──────────────────────┘  │
│   🧪 实验设计                   │                             │
│   ⚠️ 实验失败                   │   ┌─ 当前假设 ──────────┐  │
│   ✅ 实验成功                   │   │ ... 状态：部分验证 ✅  │  │
│   🔀 方向转变                   │   └──────────────────────┘  │
│   📝 章节草稿                   │                             │
│                                │   ┌─ 待办 ────────────────┐  │
│                                │   │ ☐ 补 NF-κB 抑制实验    │  │
│                                │   │ ☐ 写 Discussion       │  │
│                                │   └──────────────────────┘  │
├────────────────────────────────┴─────────────────────────────┤
│  ⚡ 快捷操作：[搜文献] [设计实验] [上传数据] [写论文] [排障]     │
└──────────────────────────────────────────────────────────────┘
```

### 5.2 交互模式

- **结构化面板**：矩阵表格、Protocol 卡片、数据图表——可视化、可操作
- **AI 对话（侧边栏）**：灵活、自由，有完整课题记忆
- **两者共享上下文**：面板上看到的 AI 都知道，AI 说的都能反映到面板

### 5.3 回溯时的关键体验

**实验失败 → 排障模式**：

- 自动关联失败实验的具体条件
- 基于文献 + 已知失败模式给出诊断
- 建议具体参数调整（浓度、时间、细胞代次等）

**结果矛盾 → 定向探索**：

- 不是空白搜索框，而是"你的结果 vs 文献结果"的对比视图
- AI 自动列出可能的解释（浓度差异、时间差异、细胞系差异等）
- 一键从解释出发重新设计实验

## 六、落地路线图（建议）

| 阶段 | 内容 | 预估周期 | 理由 |
|------|------|---------|------|
| **Phase 1：机制矩阵 MVP** | 文献检索 → 结构化提取 → 机制矩阵 + 冲突检测 + Gap Finder | 4-6 周 | 最独特、最能验证核心假设 |
| **Phase 2：实验设计助手** | 矩阵 → 实验推荐 → Protocol 生成 → 浓释计算 → 对照组检查 + 排障 | 6-8 周 | Flow 的第一段连接（文献→实验） |
| **Phase 3：论文组装器** | 矩阵→Intro、Protocol→Methods、数据→Results、自动组装 + LaTeX 输出 + Reviewer Simulator | 4-6 周 | Flow 的第二段连接（实验→论文） |
| **Phase 4：Research Memory** | 课题知识图谱、跨 session 持久化、实验室协作 | 持续迭代 | 长期护城河 |

**MVP 交付标准（Phase 1）**：研究者拿到 5 篇论文，10 分钟内看到一张结构化的机制对比表，提取准确率 >85%。

## 八、关键洞察总结

1. **机制矩阵是产品最具辨识度的功能**——"比较十篇"比"读懂一篇"更有价值
2. **三个模块串联成的 Flow 管道是护城河**——单独任何一个模块都有竞品
3. **工作流必须是循环式的**——真实科研充满失败和回溯，线性模型不成立
4. **失败是一等公民**——记录失败比记录成功更重要，它是排障和论文 Methods 的素材
5. **用得越久越有价值**——Research Memory 是长期壁垒，数据飞轮效应
